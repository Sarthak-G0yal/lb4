**Project Scope: Java Layer 4 Load Balancer for Learning**

We are building a **pure TCP Layer 4 load balancer** in **Java** as a learning project to strengthen understanding of non-blocking networking, event-driven design, TCP connection handling, byte-level forwarding, configuration mapping, and basic load-balancing behavior. The system will use a **full proxy architecture**, where the load balancer accepts a client TCP connection, opens a separate TCP connection to a backend, and forwards data between the two. The goal is not to create production-grade software, but to deeply understand how a TCP proxy works internally and how non-blocking I/O systems are structured.

We are using **Java NIO** with a **single global Selector event loop** and **one thread** managing all non-blocking I/O operations. The networking core will rely on **ServerSocketChannel**, **SocketChannel**, **Selector**, **SelectionKey**, and **ByteBuffer**. Instead of a multi-threaded boss/worker model, the entire system will be intentionally constrained to one selector-driven loop so that the control flow, state handling, and data movement remain easier to understand. Connection pairing will be handled through a custom **Session** object attached directly to each `SelectionKey`, so that client-side and backend-side state remain linked clearly inside the event loop.

We are using **one reusable Direct ByteBuffer** within the event loop to reinforce the idea of explicit buffer reuse and to keep memory behavior simple and visible. The project will focus on learning how bytes are read from one socket and written to the other, how socket readiness events drive the flow, and how the proxy reacts when a connection is closed or fails. The implementation will use a **“Brutal Teardown”** policy, meaning that on any error, EOF, or fatal socket issue, both sides of the session are immediately closed. This keeps the logic straightforward and makes the lifecycle of each session easy to reason about during learning.

For backend selection, we are replacing least-connections with **IP Hashing**. The proxy will route clients to backends based on a stable hash of the client IP, which teaches sticky routing and simple deterministic backend selection without needing atomic counters, shared load metrics, or more complex coordination. This keeps the load-balancing logic lightweight and highlights the tradeoff between simplicity and balanced distribution. The project will still support a configurable backend list so that the hash maps clients consistently across the available servers.

For configuration, we are keeping **YAML**, but we are using **Maven** and the **Jackson library** to map the YAML configuration directly into Java POJOs instead of manually parsing strings. The configuration will define the listen IP, listen port, backend host/port pairs, session and logging behavior, and the load balancing strategy. This keeps the project organized and teaches clean configuration modeling without adding unnecessary parsing complexity. The use of Maven will also help manage dependencies, build structure, and repeatable compilation.

For health handling, we are shifting to **passive health checks** in the core engine. Rather than doing active TCP probing at the start, the system will treat I/O failures, connection refusals, and socket exceptions as the primary signals that a backend or session has failed. This makes the early implementation simpler and keeps the focus on raw byte-pumping mechanics, socket errors, and session teardown behavior. Active TCP health checks are deferred to a later stage after the main proxy loop is stable and understandable.

For security and traffic control, we are **deferring active DDoS logic, SYN flood mitigation, and token-bucket rate limiting** until the later stages of the project. The early scope does not include full protection logic, because the main learning goal is to get the TCP proxy engine, selector loop, and connection handling correct first. Security controls will be considered only after the raw forwarding behavior is working reliably, so that they do not obscure the core networking lessons.

We are also **simplifying backpressure handling** by avoiding complex partial-write orchestration in the initial learning scope. The proxy will favor straightforward read-and-write loops so that the data path remains easy to follow and debug. The emphasis is on understanding how bytes move through a non-blocking TCP proxy, how sessions are created and destroyed, and how the selector wakes up on socket readiness events. Advanced write-queue management, fine-grained buffering strategies, and sophisticated flow control are intentionally avoided for this version of the project.

The main focus of the project is to learn and demonstrate how a **single-threaded NIO TCP proxy** can be built cleanly in Java, how a **session object** can represent a paired client/backend connection, how **IP-hash routing** provides deterministic backend selection, how **YAML-to-POJO mapping** can be used for clean configuration, and how **passive failure handling** works in a byte-forwarding proxy. The project is designed to make the mechanics visible and understandable, not to maximize performance or provide a feature-complete production service.

We are avoiding multi-threaded selector architectures, boss/worker event loops, least-connections counters, complex atomic coordination, active health probes in the core engine, rate limiting in the early implementation, advanced DDoS/SYN flood mitigation, UDP support, HTTP parsing, TLS termination, distributed clustering, and production-grade backpressure optimization. We are also avoiding frameworks that hide the networking internals, because the purpose of the project is to build understanding of the low-level mechanics using the Java standard library plus lightweight configuration tooling.

The final project should be a clean, educational Java codebase that shows a complete non-blocking TCP proxy flow from accept to forward to teardown, with clear Session handling, deterministic backend routing, YAML-based configuration, passive error-based backend failure behavior, and a deliberately simple event-loop design that makes the networking logic easy to study, debug, and explain.
